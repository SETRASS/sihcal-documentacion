This page provides an overview of the deployment infrastructure, containerization strategy, and hosting environment for the SETRASS ECMS Web UI application. It focuses on the technical aspects of how the application is built, deployed, and served to users across different environments. For information about the application's architectural components, see Architecture.

Container-Based Deployment Model

The SETRASS ECMS Web UI is deployed using a container-based model with Docker and Kubernetes. This approach ensures consistency across environments and facilitates scalable deployment.

The application uses a single-container model based on Nginx, where the built Angular application is deployed inside an Nginx container. This approach provides a lightweight and efficient deployment solution.

Kubernetes Deployment

The application is deployed to a Kubernetes cluster, which provides orchestration, scaling, and service discovery. The deployment configuration is defined in the k8s.yml file.

Deployment Configuration

The Kubernetes configuration includes:

  1. Deployment: Named setrass-ecms-web-ui with change-cause annotations for tracking updates
  2. ReplicaSet: Maintains 2 replicas of the application for high availability
  3. Pod Template: Defines the container specifications
  4. Service: Named setrass-ecms-web-ui-svc that exposes the application on port 80

CI/CD Pipeline with Jenkins

The application uses a Jenkins pipeline for automated building, testing, and deployment. The pipeline is defined in the Jenkinsfile.

The Jenkins pipeline consists of two main stages:

  1. Build dist:

    • Installs dependencies with npm
    • Builds the Angular application
    • Stashes the build artifacts for the next stage

  2. Build Docker, Save in ECR and Publish K8s:

    • Builds a Docker image containing the Angular application
    • Pushes the image to Amazon ECR (Elastic Container Registry)
    • Deploys the application to the appropriate Kubernetes environment

Environment-Specific Deployment

The pipeline includes logic for deploying to different environments:

  • For the development environment:

    • Uses AWS credentials to update the EKS cluster configuration
    • Applies the Kubernetes manifest to the development namespace

  • For other environments (staging, production):

    • Uses a configuration file with specific cluster settings
    • Applies the Kubernetes manifest to the specified namespace

Sources: Jenkinsfile

Version Management

The application version is maintained in multiple locations to ensure consistency:

  1. package.json: Defines the application version (currently 2.0.88)
  2. Jenkinsfile: Sets the DEPLOY_VERSION environment variable
  3. k8s.yml: References the specific image version to deploy
  4. Footer Component: Displays the version to users
  5. Web Manifest: Includes version in the PWA short name

This approach ensures that the version is consistent across all deployment artifacts and visible to users.

Web Server Configuration

The application is served using Nginx, which provides:

  1. Static file serving for the Angular application
  2. URL rewriting for Angular's client-side routing
  3. Compression and caching optimizations

Progressive Web App Support

The SETRASS ECMS Web UI is configured as a Progressive Web App (PWA), which provides enhanced user experience through features like offline support, installability, and improved performance.

The PWA configuration includes:

  1. Web Manifest: Defines the application name, icons, and display properties
  2. Service Worker: Provided by Angular's @angular/service-worker package
  3. Icons: Multiple sizes available for different devices and contexts

Environment Configuration System

The application supports multiple deployment environments through Angular's environment configuration system. While the actual environment files aren't shown in the provided source code, the Jenkins pipeline and Kubernetes configuration indicate support for at least:

  • Development environment
  • Staging environment
  • Production environment

Each environment can have different API endpoints, feature flags, and configuration values.

Dependencies and Third-Party Libraries

The application relies on a substantial set of dependencies, which are managed through npm. These include:

  1. Angular Core Libraries: Version 13.3.0
  2. UI Component Libraries: Including Bootstrap, Angular Material, and custom SETRASS components
  3. Utility Libraries: For charts, forms, dates, and other functionality
  4. PWA Support: Through Angular service worker

The complete list of dependencies is extensive and can be found in the package.json file.
